# Configuration Reference

SODA uses three configuration layers, each with its own discovery order:

- **`soda.yaml`** — project configuration (ticket source, model, budget, repos). Loaded once at startup.
- **`phases.yaml`** — pipeline definition (phase names, tools, timeouts, retry policies). Loaded per run.
- **Prompt overrides** — per-phase Markdown templates that replace embedded defaults. Resolved per phase.

---

## Config Discovery

SODA looks for `soda.yaml` in the following order (first found wins):

1. `--config <path>` flag (explicit path, always wins)
2. `soda.yaml` in the current working directory (project-local, generated by `soda init`)
3. `~/.config/soda/config.yaml` (user-global fallback)

Generate a starter config with `soda init`, then edit it before running the pipeline.

## soda.yaml Reference

### Ticket source

```yaml
ticket_source: github   # required: "github" or "jira"
```

#### GitHub Issues

```yaml
github:
  owner: string          # repository owner or org (e.g. "myorg")
  repo: string           # repository name (e.g. "my-service")
  fetch_comments: bool   # fetch issue comments for spec/plan extraction (default: false)
  spec:
    start_marker: string # marker that begins the spec block (e.g. "<!-- spec:start -->")
    end_marker: string   # marker that ends the spec block   (e.g. "<!-- spec:end -->")
  plan:
    start_marker: string # marker that begins the plan block (e.g. "<!-- plan:start -->")
    end_marker: string   # marker that ends the plan block   (e.g. "<!-- plan:end -->")
```

When `fetch_comments` is `true`, SODA scans issue comments for content between the configured start/end markers. If multiple comments contain the same markers, the last one wins. The extracted content is injected as `ExistingSpec` / `ExistingPlan` into prompt templates, and triage sets `skip_plan: true` when a plan is found.

#### Jira

```yaml
jira:
  command: string        # CLI binary to use (default: "wtmcp")
  project: string        # Jira project key (e.g. "MYPROJECT")
  query: string          # JQL query to filter workable issues (optional)
  extraction:
    spec_field: string   # custom field ID containing the spec (e.g. "customfield_10100")
    plan_field: string   # custom field ID containing the plan (e.g. "customfield_10101")
    subtask_field: string # custom field ID for subtask references (optional)
    spec:
      start_marker: string
      end_marker: string
    plan:
      start_marker: string
      end_marker: string
```

Multiple extraction strategies are tried in order — description markers, then custom fields. The first strategy that finds content wins.

### Authentication

SODA resolves Claude Code credentials using the following precedence chain
(first match wins):

1. **Sandbox proxy** (`sandbox.proxy.enabled: true`) — the proxy injects credentials on the host side; no API key needed inside the sandbox.
2. **`ANTHROPIC_API_KEY`** environment variable — the simplest method for direct API access.
3. **Vertex AI** (`CLAUDE_CODE_USE_VERTEX` env var) — uses Google Cloud Application Default Credentials.
4. **`auth.api_key_helper`** — a script that prints an API key to stdout, useful for keychain/vault integration.
5. **Claude Code login** — if none of the above are set, Claude Code falls back to its own `claude login` session.

```yaml
auth:
  api_key_helper: string   # path to a script/binary that prints an API key to stdout (optional)
```

When `api_key_helper` is set, SODA writes a temporary Claude Code settings file with `apiKeyHelper` pointing to the configured script. Claude Code invokes the script before each API call to obtain a fresh key. This avoids storing long-lived keys in environment variables.

Example helper script (`~/.local/bin/get-claude-key`):

```bash
#!/bin/bash
# Fetch API key from macOS Keychain
security find-generic-password -s "anthropic-api-key" -w
```

```yaml
auth:
  api_key_helper: ~/.local/bin/get-claude-key
```

Run `soda doctor` to verify your authentication setup — the `claude-auth` check
reports which method was detected.

### Model

```yaml
model: string   # default: "claude-sonnet-4-20250514"
```

Global default model used for all phases. Individual phases can override this with their own `model` field in `phases.yaml`. This value is compiled into the binary as a default; override it in `soda.yaml` at runtime.

### Sandbox

The sandbox requires CGO — builds with `CGO_ENABLED=0` use a stub that returns an error at runtime.

```yaml
sandbox:
  enabled: bool        # enable OS-level isolation via go-arapuca (default: false)
  binary: string       # sandbox binary name (optional; resolved automatically)
  limits:
    memory_mb: int     # cgroup memory limit in MiB (default: 2048)
    cpu_percent: int   # cgroup CPU quota as a percentage (default: 200)
    max_pids: int      # cgroup PID limit (default: 256)
  proxy:
    enabled: bool             # route API calls through a host-side proxy (default: false)
    upstream_url: string      # upstream API URL (default: ANTHROPIC_BASE_URL or https://api.anthropic.com)
    max_input_tokens: int     # per-session input token budget; 0 = unlimited (default: 0)
    max_output_tokens: int    # per-session output token budget; 0 = unlimited (default: 0)
    log_dir: string           # request/response audit log directory; empty = no logging
```

When the proxy is enabled, API calls are bridged through a Unix socket for credential isolation, token metering, and budget enforcement. Requires `sandbox.enabled: true`.

### Budget

```yaml
limits:
  max_cost_per_ticket: float       # hard USD cap across the full pipeline run; 0 = no limit (default: 0)
  max_cost_per_phase: float        # cumulative USD cap per phase across rework cycles; 0 = no limit (default: 0)
  max_cost_per_generation: float   # per-attempt USD cap (resets each generation); 0 = disabled (default: 0)
  max_pipeline_duration: string    # max wall-clock time for the full pipeline (e.g. "2h", "90m"); empty = no limit
  max_api_concurrency: int         # max concurrent API calls, e.g. parallel reviewers; 0 = unlimited (default: 0)
  max_diff_bytes: int              # max bytes of git diff injected into rework prompts; 0 = use default 50000 (default: 0)
  max_sibling_context_bytes: int   # max bytes of sibling-function context injected into implement; 0 = use default 20000 (default: 0)
```

**Note on `max_cost_per_phase`:** This cap is cumulative — it accumulates across rework and corrective cycles for the same phase, not just per individual generation. Use `max_cost_per_generation` to cap per-attempt costs instead.

### Execution

```yaml
mode: string         # "autonomous" (run without pausing) or "checkpoint" (pause after each phase) (default: "autonomous")
worktree_dir: string # directory for git worktrees (default: ".worktrees")
state_dir: string    # directory for pipeline state and logs (default: ".soda")
phases_path: string  # explicit path to a phases.yaml file; overrides CWD discovery for the default pipeline
prompts_path: string # base directory for prompt overrides; searched before CWD and embedded prompts
```

### Context injection

Files listed under `context` are read from disk and injected into every phase's system prompt. Use this for project-wide conventions (e.g. `AGENTS.md`).

```yaml
context:
  - AGENTS.md
  - docs/conventions.md
```

Files listed under `phase_context` are injected only into the named phase's system prompt:

```yaml
phase_context:
  implement:
    - docs/gotchas.md
  verify:
    - docs/gotchas.md
```

### Monitor

```yaml
monitor:
  self_user: string           # GitHub username of the bot/agent (required for comment classification)
  bot_users: [string]         # known bot usernames to filter out (e.g. "dependabot", "renovate")
  profile: string             # behavior preset: "conservative", "smart", or "aggressive"
  codeowners: string          # path to CODEOWNERS file for authority resolution (optional)
```

**`self_user` is required** for the monitor phase to distinguish self-authored comments from external ones. Without it, the monitor cannot classify comments and falls back to a stub.

Monitor profiles:

| Profile | Poll interval | Response rounds | Auto-fix nits | Auto-rebase | Non-auth comments |
|---------|--------------|-----------------|--------------|-------------|-------------------|
| `conservative` | 5m | 2 | no | no | ignore |
| `smart` (default) | 2m | 3 | yes | yes | ignore |
| `aggressive` | 1m | 5 | yes | yes | respond |

Fine-grained polling settings (`initial_interval`, `max_response_rounds`, etc.) are configured in `phases.yaml` under the `monitor` phase's `polling` block.

---

### Repos

Multi-repo support: define one entry per repository that SODA may create pull requests against.

```yaml
repos:
  - name: string         # repo identifier (used in prompts and logs)
    forge: string        # "github" or "gitlab"
    push_to: string      # owner/repo to push branches to (e.g. "myuser/my-service")
    target: string       # owner/repo to open PRs against (e.g. "myorg/my-service")
    description: string  # short description injected into prompts (optional)
    formatter: string    # format command run before commit (e.g. "gofmt -w .")
    test_command: string # test command run during verify (e.g. "go test ./...")
    labels: [string]     # labels applied to created PRs (e.g. ["ai-assisted"])
    trailers: [string]   # git commit trailers appended to messages (e.g. ["Assisted-by: SODA"])
```

The first entry in `repos` is used as the default when no other matching repo is found for the ticket's target.

## phases.yaml Reference

The default pipeline is defined in `phases.yaml`. SODA looks for this file in the following order:

1. `phases_path` from `soda.yaml` (if set; used as-is for the default pipeline)
2. `phases.yaml` in the current working directory
3. `~/.config/soda/phases.yaml`
4. Embedded default (compiled into the binary)

Named pipelines use the filename `phases-<name>.yaml` and follow the same search order (without the `phases_path` override).

### Phase fields

```yaml
phases:
  - name: string              # phase identifier; used in logs, state files, and cross-references
    type: string              # phase type (see Phase types below); omit for normal phases
    prompt: string            # path to the prompt template, relative to the phases.yaml directory
                              # (e.g. "prompts/implement.md")
    schema: string            # structured output schema: inline JSON, file path, or empty to use
                              # the auto-generated schema for the phase name
    model: string             # per-phase model override; empty uses the global model from soda.yaml
    tools: [string]           # allowed tools list (see Tool scoping below)
    timeout: duration         # phase timeout as a Go duration string (e.g. "25m", "3m")
    retry:
      transient: int          # retries for transient API errors (rate limits, timeouts) (default: 2)
      parse: int              # retries when output fails JSON schema validation (default: 1)
      semantic: int           # retries when output is structurally valid but semantically wrong (default: 1)
    depends_on: [string]      # phase names that must complete before this phase runs
    feedback_from: [string]   # phase names whose output is injected as rework feedback into this phase's prompt
```

### Phase types

| Type | Behaviour |
|------|-----------|
| *(omitted)* | Normal phase — runs once in forward order |
| `corrective` | Corrective phase — only runs when triggered by another phase's `corrective` config block |
| `post-submit` | Runs after submit; failures are swallowed (best-effort, non-terminal) |
| `parallel-review` | Runs multiple reviewers in parallel; results are merged into a single output |
| `polling` | Long-running polling loop (used for the monitor phase) |

### Rework routing

To route back to an earlier phase on a bad verdict, add a `rework` block to the phase that produces the verdict:

```yaml
    rework:
      target: string    # phase name to route back to when this phase returns a "rework" verdict
```

Example: the review phase routes back to implement on a `rework` verdict.

### Corrective routing

To trigger a corrective phase on failure, add a `corrective` block to the triggering phase:

```yaml
    corrective:
      phase: string        # corrective phase name to invoke on failure (e.g. "patch")
      max_attempts: int    # max corrective cycles before exhaustion policy applies
      on_exhausted: string # what to do when max_attempts is reached: "stop", "escalate", or "retry"
      escalate_to: string  # target phase for "escalate" policy (e.g. "implement")
```

Example: the verify phase triggers the patch corrective phase on a FAIL verdict.

### Parallel review

For `type: parallel-review`, configure individual reviewers and quorum requirements:

```yaml
    min_reviewers: int         # minimum number of reviewers that must succeed; 0 means all must succeed
    reviewer_stagger: duration # delay between starting each reviewer (optional; reduces API burst)
    reviewers:
      - name: string           # reviewer identifier (used in log filenames and output)
        prompt: string         # reviewer-specific prompt template path
        focus: string          # short description of the reviewer's focus area (injected into the prompt)
        model: string          # per-reviewer model override (optional)
```

### Polling (monitor)

For `type: polling`, add a `polling` block:

```yaml
    polling:
      initial_interval: duration    # time between poll cycles (e.g. "2m")
      max_interval: duration        # maximum poll interval after escalation (e.g. "5m")
      escalate_after: duration      # how long before switching to max_interval (e.g. "30m")
      max_duration: duration        # total wall-clock limit for the monitor phase (e.g. "4h")
      max_response_rounds: int      # max Claude sessions for comment responses (counts fix + reply rounds)
      respond_to_comments: bool     # enable comment classification and response (requires self_user in soda.yaml)
      auto_merge: bool              # merge PR automatically when CI is green and approved (default: false)
```

### Tool scoping

The `tools` list in each phase is passed to Claude Code as `--allowed-tools`. This is advisory — use sandbox isolation for enforcement in unattended runs.

Supported tool names and patterns:

| Tool | Description |
|------|-------------|
| `Read` | Read files |
| `Write` | Write files |
| `Edit` | Edit files in-place |
| `Glob` | File pattern matching |
| `Grep` | Content search |
| `Bash` | Unrestricted shell access |
| `Bash(git:*)` | Git subcommands only (`git log`, `git diff`, `git commit`, …) |
| `Bash(gh:*)` | GitHub CLI (`gh pr create`, `gh issue list`, …) |
| `Bash(glab:*)` | GitLab CLI |
| `Bash(ls:*)` | Directory listing only |
| `Bash(go test:*)` | Go test invocations only |

Combine entries to grant a specific set of capabilities. Examples from the default pipeline:

```yaml
# Triage and plan — read-only + limited shell
tools:
  - Read
  - Glob
  - Grep
  - Bash(git:*)
  - Bash(ls:*)

# Implement and patch — full access
tools:
  - Read
  - Write
  - Edit
  - Glob
  - Grep
  - Bash

# Submit — forge CLI only
tools:
  - Bash(git:*)
  - Bash(gh:*)
  - Bash(glab:*)

# Follow-up — GitHub CLI only
tools:
  - Bash(gh:*)
```

## Named Pipelines

### Discovery order

SODA resolves named pipelines (`phases-<name>.yaml`) in this order (first found wins):

1. Current working directory (`./phases-<name>.yaml`)
2. User config directory (`~/.config/soda/phases-<name>.yaml`)
3. Embedded defaults (compiled into the binary)

The default pipeline (`phases.yaml`) also checks `phases_path` in `soda.yaml` before the CWD.

### Built-in pipelines

| Name | Phases | Use case |
|------|--------|----------|
| `default` | triage → plan → implement → verify → review → submit → monitor | Full pipeline for standard tickets |
| `quick-fix` | implement → verify → submit | Small, well-understood fixes that don't need triage or planning |
| `docs-only` | plan → implement → submit | Documentation-only changes (Sonnet) |

### Listing available pipelines

```bash
soda pipelines
```

This lists all discovered pipelines from the working directory, user config directory, and embedded defaults, along with their source and file path.

### Creating a custom pipeline

```bash
soda pipelines new <name>              # write phases-<name>.yaml to current directory
soda pipelines new <name> --global     # write to ~/.config/soda/phases-<name>.yaml
```

This scaffolds a new pipeline definition file named `phases-<name>.yaml`. Edit the generated file to add, remove, or reorder phases.

Custom pipelines follow the same file format as the default `phases.yaml`. The pipeline file is self-contained: it references prompt templates and the schema is either inline JSON or resolved automatically from the phase name.

### Running with a named pipeline

```bash
soda run <ticket> --pipeline quick-fix
soda run <ticket> --pipeline my-custom-pipeline
```

## Prompt Overrides

Each phase renders a Go template (`text/template`) to produce its system prompt. SODA resolves prompt templates in this order (first found wins):

1. `~/.config/soda/prompts/<phase>.md` — user-global override
2. `<prompts_path>/prompts/<phase>.md` — `prompts_path` from `soda.yaml` (if set)
3. `./prompts/<phase>.md` — project-local override (current working directory)
4. Embedded default (compiled into the binary)

> **Note:** The `prompts/` subdirectory is part of the path because `phases.yaml` references prompts as `prompts/<phase>.md`. Your override files must mirror this structure under whatever base directory you use.

### File naming

Phase prompts use the pattern `prompts/<phase>.md`, where `<phase>` matches the `name` field in `phases.yaml`. Examples:

- `prompts/implement.md`
- `prompts/triage.md`
- `prompts/verify.md`
- `prompts/review-go.md` (reviewer-specific prompts use `review-<reviewer-name>.md`)

### Template variables

All prompt templates receive a `PromptData` struct with the following fields:

| Variable | Type | Description |
|----------|------|-------------|
| `{{.Ticket.Key}}` | string | Ticket identifier (e.g. "PROJ-123") |
| `{{.Ticket.Summary}}` | string | Ticket title |
| `{{.Ticket.Description}}` | string | Full ticket description |
| `{{.Ticket.Type}}` | string | Ticket type (e.g. "bug", "feature") |
| `{{.Ticket.Priority}}` | string | Ticket priority |
| `{{.Ticket.AcceptanceCriteria}}` | []string | Acceptance criteria list |
| `{{.Ticket.Comments}}` | []TicketComment | Issue comments (when `fetch_comments: true`) |
| `{{.Ticket.ExistingSpec}}` | string | Extracted spec content (if found) |
| `{{.Ticket.ExistingPlan}}` | string | Extracted plan content (if found) |
| `{{.Artifacts.Triage}}` | string | Triage phase JSON output |
| `{{.Artifacts.Plan}}` | string | Plan phase JSON output |
| `{{.Artifacts.Implement}}` | string | Implement phase JSON output |
| `{{.Artifacts.Verify}}` | string | Verify phase JSON output |
| `{{.Artifacts.Review}}` | string | Review phase JSON output |
| `{{.Artifacts.Patch}}` | string | Patch phase JSON output |
| `{{.Artifacts.Submit.PRURL}}` | string | PR URL from submit phase |
| `{{.Artifacts.Extras}}` | map[string]string | Artifacts from custom phases, keyed by phase name |
| `{{.Config.Repos}}` | []RepoConfig | All configured repos |
| `{{.Config.Repo}}` | RepoConfig | Primary repo for this ticket |
| `{{.Config.Formatter}}` | string | Format command from primary repo |
| `{{.Config.TestCommand}}` | string | Test command from primary repo |
| `{{.Context.ProjectContext}}` | string | Contents of `context` files (concatenated) |
| `{{.DetectedStack.Language}}` | string | Auto-detected language (e.g. "go", "python") |
| `{{.DetectedStack.Forge}}` | string | Auto-detected forge (e.g. "github", "gitlab") |
| `{{.DetectedStack.Owner}}` | string | Repository owner |
| `{{.DetectedStack.Repo}}` | string | Repository name |
| `{{.WorktreePath}}` | string | Absolute path to the git worktree |
| `{{.Branch}}` | string | Working branch name |
| `{{.BaseBranch}}` | string | Base branch (e.g. "main") |
| `{{.ReworkFeedback}}` | *ReworkFeedback | Feedback from a previous failed cycle (nil on first run) |
| `{{.DiffContext}}` | string | Git diff of the current branch vs base (injected for monitor and corrective phases) |
| `{{.SiblingContext}}` | string | Function signatures from files referenced in the plan |

Use `{{if .ReworkFeedback}}` guards for sections that only apply on rework cycles.

The built-in template function `{{add $a $b}}` is available for 1-based display indexing.

### Example: project-specific implement prompt

Create `./prompts/implement.md` in your project root with your project-specific instructions prepended to the standard prompt. SODA validates override templates before using them — if the template has syntax errors or references unknown fields, SODA falls back to the embedded default and logs the reason.

```markdown
# Implement Phase

## Project-specific conventions

- All database queries must use the `db.QueryContext` helper, never `db.Query`
- Error messages must be wrapped with `fmt.Errorf("package.function: %w", err)`
- New packages require an entry in `docs/packages.md`

## Task

{{.Ticket.Summary}}

...rest of standard prompt...
```

Run `soda render-prompt --phase implement --ticket <key>` to preview the rendered output before executing the pipeline.
