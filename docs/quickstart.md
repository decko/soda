# Quickstart

This guide walks you from a fresh machine to a submitted pull request using SODA.

> **Cost note:** A typical SODA run costs **$5–11 USD** in Claude API usage. Set
> `max_cost_per_ticket` in `soda.yaml` to cap spending before you start.
> See [Budget cap](#budget-cap) below.

---

## Prerequisites

Before starting, make sure you have each of the following.

### 1. Claude Code CLI with an active subscription

SODA drives Claude Code as its agent. You need the CLI installed **and** an active
Anthropic subscription — the free tier does not support the API usage SODA requires.

- Sign up: <https://console.anthropic.com>
- Install Claude Code: <https://docs.anthropic.com/en/docs/claude-code>
- Minimum version: **2.1.81** (SODA uses `--bare`, introduced in that release)

Verify your installation:

```
claude --version
```

**Authentication:** SODA needs a way to authenticate with the Anthropic API.
The simplest method is to set the `ANTHROPIC_API_KEY` environment variable:

```bash
export ANTHROPIC_API_KEY=sk-ant-...
```

For keychain or vault-based workflows, you can configure `auth.api_key_helper`
in `soda.yaml` instead — see [Configuration: Authentication](configuration.md#authentication).

### 2. `gh` CLI — installed and authenticated

SODA uses `gh` to fetch GitHub issues and create pull requests.

- Install: <https://cli.github.com>
- Authenticate: `gh auth login`

### 3. Git 2.9+

Required for `core.hooksPath` support and worktree operations.

```
git --version
```

### 4. Go 1.25+ (if installing via `go install`)

Only needed for the `go install` installation method. If you download a binary
from the releases page, you can skip Go.

```
go version
```

### 5. A project with a GitHub-hosted repo

This guide uses a Go project as the example, but SODA works with any language
that `soda init` can detect (Go, Python, TypeScript, Rust, Java). You need:

- A GitHub repository (public or private)
- Issues enabled
- At least one issue to run against

### 6. A GitHub issue to work on

Pick an existing open issue or create one. Note the issue number — you will
pass it to `soda run`.

## Step 1 — Install SODA

**Option A: `go install`** (requires Go 1.25+)

```bash
go install github.com/decko/soda/cmd/soda@latest
```

**Option B: Download a binary**

Download the latest release for your platform from the
[releases page](https://github.com/decko/soda/releases), unzip it, and
move the `soda` binary somewhere on your `PATH`.

Verify the install:

```
soda version
```

---

## Step 2 — Navigate to your project

```bash
cd /path/to/your-go-project
```

SODA must run from the root of your repository — the directory that contains
`.git/`. It creates worktrees at `.worktrees/soda/<ticket-key>/` and stores
session state in `.soda/<ticket-key>/`.

---

## Step 3 — Run `soda init`

```bash
soda init
```

`soda init` scans the repository and writes a starter `soda.yaml` tailored to
your project. It auto-detects:

- Language and tooling (Go modules, test runner, formatter)
- Forge (GitHub vs GitLab)
- Repository name and owner

**Sample output:**

```
Detected: Go project (github.com/myorg/my-service)  # example; your language may differ
Forge:     github
Writing:   soda.yaml
Writing:   .gitignore entries for .soda/ and .worktrees/
Done. Edit soda.yaml before running the pipeline.
```

Open `soda.yaml` and review the generated configuration. At minimum, check:

- `github.owner` and `github.repo` match your repository
- `model` is the Claude model you want to use (default: `claude-opus-4-6`)
- `limits.max_cost_per_ticket` is set to a comfortable ceiling

See [`config.example.yaml`](../config.example.yaml) for the full annotated
reference.

### Budget cap

To cap spending before your first run, add or adjust this block in `soda.yaml`:

```yaml
limits:
  max_cost_per_ticket: 15.00   # hard stop for the whole pipeline
  max_cost_per_phase: 8.00     # per-phase ceiling (prevents runaway rework)
```

SODA stops with a `BudgetExceededError` if either limit is hit. The default
`max_cost_per_ticket` generated by `soda init` is `30.00`; lower it if you
want a tighter guardrail.

> **Tip — check prompt sizes first:** Before your first real run, use
> `soda run <issue> --estimate` to see per-phase token estimates without
> executing anything. If any phase exceeds ~60,000 tokens, consider splitting
> the ticket. See [`docs/troubleshooting.md`](troubleshooting.md#11-token-estimation-and-prompt-sizing)
> for details.

## Step 4 — Run `soda doctor`

```bash
soda doctor
```

`soda doctor` checks that every required tool is present and your environment
is ready to run the pipeline.

**Sample output (all passing):**

```
✓ git: /usr/bin/git
✓ git-repo: inside a git repository
✓ claude: /usr/local/bin/claude
✓ claude-version: 2.1.81
✓ config: soda.yaml (local)
✓ config-valid: soda.yaml parses successfully
✓ claude-auth: ANTHROPIC_API_KEY is set
✓ gh: /usr/local/bin/gh
✓ gh-auth: authenticated
⚠ node: not found in PATH (optional, needed only for sandboxed execution)

All checks passed
```

Symbols:
- `✓` — required check passed
- `⚠` — optional check not satisfied (the pipeline can still run)
- `✗` — required check failed (fix before proceeding)

Each `✗` line includes a `fix:` hint. Fix all required failures before
moving on.

---

## Step 5 — Pick or create a GitHub issue

Browse your repository's issues and pick one that is:

- Well-described (clear acceptance criteria help the agent)
- Self-contained (no external API calls, no database migrations)
- Reasonable in scope (aim for < 500 lines of change on your first run)

Note the issue number. For example, if the URL is
`https://github.com/myorg/my-service/issues/42`, the issue number is `42`.

If you want to create a test issue:

```bash
gh issue create \
  --title "Add a hello-world HTTP handler" \
  --body "Add a /hello endpoint that returns {\"message\": \"hello\"} with status 200. Include a test."
```

---

## Step 6 — Run `soda run <issue-number>`

```bash
soda run 42
```

SODA opens a TUI showing each phase as it runs. You can watch the live output
stream from Claude in the bottom panel, and track phase status and cost in the
side panel.

### What happens phase by phase

| Phase | What SODA does | Typical duration |
|-------|---------------|-----------------|
| **Triage** | Reads the issue and codebase; classifies the ticket and identifies which files to change | 1–3 min |
| **Plan** | Designs an approach and breaks it into atomic tasks | 2–5 min |
| **Implement** | Writes code, runs tests, commits | 5–25 min |
| **Verify** | Runs your test suite and checks acceptance criteria | 2–8 min |
| **Review** | Specialist agents review the code for correctness and style | 3–12 min |
| **Submit** | Pushes the branch and opens a pull request | < 1 min |
| **Monitor** | Polls the PR for review comments and responds | ongoing |

If **Verify** fails, SODA runs a corrective **Patch** phase automatically
before re-verifying. If **Review** finds critical issues, it routes back to
**Implement** for rework (up to 2 cycles).

### Sample TUI output

```

  ─────────────────────────│──────────────────────────────────
```

## Step 7 — Find your pull request

When Submit completes, SODA prints the PR URL:

```
PR created: https://github.com/myorg/my-service/pull/43
```

You can also find it with:

```bash
gh pr list
```

The PR:
- Is opened against `main` from a branch named `soda/<issue-number>-<slug>`
- References the original issue in the title
- Includes an `Assisted-by: SODA` trailer in the commit message

Review the diff, request changes if needed, and merge when satisfied.

## What to do if it fails

SODA surfaces failures in the TUI with the failing phase highlighted and an
error message. The most common failure modes and their fixes are in
[`docs/troubleshooting.md`](troubleshooting.md).

Quick checklist:

| Symptom | Likely cause | Fix |
|---------|-------------|-----|
| `BudgetExceededError` | Per-ticket or per-phase cap hit | Raise `max_cost_per_ticket` in `soda.yaml` |
| `PhaseGateError` at triage | Issue classified as not automatable | Rewrite the issue with clearer acceptance criteria |
| `PhaseGateError` at verify | Tests fail and patch exhausted | Check `soda history 42 --detail` for the verify output |
| `claude: command not found` | Claude Code not installed | Run `soda doctor` and follow the fix hint |
| `gh: authentication required` | `gh` not authenticated | Run `gh auth login` |
| Pipeline exits mid-run | Transient API error | Re-run with `soda run 42 --from last` to resume |

### Resuming a failed run

If the pipeline exits before Submit, you can resume from the last failed phase
without re-running earlier phases:

```bash
soda run 42 --from last
```

Or resume from a specific phase:

```bash
soda run 42 --from implement
```

Session state is preserved in `.soda/42/` between runs.

## Try it on SODA itself

The fastest way to test your setup end-to-end is to run SODA on one of its own
issues:

1. **Fork the SODA repository** on GitHub.
2. **Clone your fork** and follow this guide from Step 3.
3. In `soda.yaml`, set `github.owner` to your GitHub username and
   `github.repo` to `soda`.
4. Browse issues labeled [`good-first-issue`](https://github.com/decko/soda/labels/good-first-issue)
   and pick one.
5. Run `soda run <issue-number>`.

SODA is implemented using SODA — the repository is self-describing and the
issues follow the structured format the agent expects.
